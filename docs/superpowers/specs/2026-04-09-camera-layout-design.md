# 摄像头平面图自动布点系统设计

## 1. 背景与目标

目标是开发一套本地原型系统，支持上传门店或建筑物平面图后，自动识别图纸比例与结构信息，依据摄像头覆盖距离与类型自动估算所需摄像头数量，并在图纸上标注摄像头点位与覆盖范围。系统还需要支持人工微调点位、方向和覆盖参数，并最终导出适合交付客户的标注图。

该系统的使用场景是门店监控设计交付，输出成果需要兼顾业务可读性与后续可编辑性。

## 2. 用户确认的约束

- 图纸比例优先通过自动识别比例尺或尺寸标注获取，识别失败时允许人工校准。
- 第一版以本地原型工具形式实现。
- 同时支持方向型摄像头和全景型摄像头。
- 自动生成点位后允许人工拖拽、旋转、增删。
- 上传格式支持 `JPG`、`PNG`、`PDF`。
- 自动布点时考虑墙体/隔断遮挡，覆盖不能穿墙，门洞或开口允许通过。
- 成果导出支持标注后的 `PNG/JPG` 和 `PDF`。

## 3. 总体方案

系统采用“本地浏览器原型 + 本地算法服务”的形式运行。用户在浏览器中完成图纸上传、参数调整、自动布点、人工微调和导出，所有识别与计算均在本机完成，不依赖云端服务。

推荐技术栈如下：

- 前端：`React + TypeScript + Vite`
- 图形画布：`react-konva` 或等价 Canvas 封装
- 后端：`Python + FastAPI`
- 图像处理：`OpenCV`
- OCR 与文本识别：优先接入本地 OCR 能力，第一版可先采用成熟 Python OCR 库
- 导出：前端导出图片，后端负责 PDF 合成与项目文件保存

选择这套技术栈的原因：

- 前端画布交互和拖拽旋转编辑适合用 React + Canvas
- 图纸识别、遮挡检测和 OCR 在 Python 生态里更容易落地
- 本地浏览器原型可同时兼顾交互效率与后续扩展能力

整体拆分为四层：

1. 前端交互层
   - 上传图纸
   - 展示平面图底图
   - 叠加摄像头点位与覆盖图形
   - 提供覆盖距离滑动条
   - 提供人工编辑交互
   - 执行导出

2. 图纸解析层
   - 解析 `JPG/PNG/PDF`
   - 自动识别比例尺和尺寸标注
   - 提取墙体、隔断、门洞
   - 在识别不准时回退到人工修正流程

3. 布点计算层
   - 基于摄像头类型、覆盖距离、视场角和遮挡规则自动生成建议点位
   - 估算建议摄像头数量
   - 计算盲区与重复覆盖区域

4. 导出与项目持久化层
   - 导出标注图为 `PNG/JPG/PDF`
   - 保存本地项目文件以供再次编辑

## 4. 核心交互流程

推荐采用单页工作台布局：

- 左侧为流程导航：上传图纸 -> 比例识别 -> 墙体/门洞提取 -> 自动布点 -> 人工微调 -> 导出
- 中央为图纸画布：展示底图、墙体、门洞、摄像头点位与覆盖区域
- 右侧为参数与属性面板：覆盖距离滑块、视场角、遮挡规则、自动计算结果、当前选中摄像头属性
- 底部或侧边展示摘要：建议摄像头数量、覆盖率、盲区提示、重叠提示

推荐默认工作流程：

1. 上传图纸
2. 自动识别比例尺或尺寸标注
3. 自动识别墙体和门洞
4. 设置覆盖距离和摄像头类型
5. 自动计算建议点位与数量
6. 人工微调点位、方向与结构修正
7. 导出交付图或保存项目文件

## 5. 自动识别设计

### 5.1 图纸预处理

- 对上传图像做清晰化、边缘增强、对比度调整与二值化
- PDF 优先渲染为高分辨率位图后再进入后续识别流程

### 5.2 比例识别

目标是将图纸中的像素距离换算为真实长度。

优先级如下：

1. 自动识别比例尺图形
2. 自动识别尺寸标注文本与对应尺寸线
3. 若识别置信度不足，允许人工画一条已知长度线并输入真实长度完成校准

系统需要记录识别来源和置信度，便于提示用户是否需要人工确认。

### 5.3 结构识别

第一版重点不是完整 CAD 级重建，而是识别布点相关的核心结构：

- 外墙与内墙
- 隔断
- 门洞或开口
- 可通行区域

系统应当允许用户对识别结果进行修正，包括补墙、删墙、标记门洞和修正开口。

## 6. 自动布点算法设计

第一版推荐采用“栅格覆盖 + 候选点生成 + 贪心选点 + 遮挡检测”的方案，而不是高复杂度全局最优求解。原因是该方案更容易稳定落地，且与人工修正配合良好。

### 6.1 支持的摄像头模型

1. 方向型摄像头
   - 覆盖区域为扇形
   - 参数包括位置、朝向、覆盖距离、视场角

2. 全景型摄像头
   - 覆盖区域为圆形
   - 参数包括位置与覆盖距离

### 6.2 计算目标

布点目标不是单纯最少数量，而是综合优化以下因素：

- 可覆盖面积最大化
- 盲区最小化
- 重复覆盖不过高
- 点位尽量符合常见安装习惯，例如靠边、靠角、靠通道节点
- 覆盖不能穿墙
- 覆盖可以跨门洞/开口传播

### 6.3 推荐计算流程

1. 将可布点区域与需覆盖区域映射为二维栅格
2. 基于墙线、角点、边界和通道生成候选安装点
3. 对每个候选点模拟不同摄像头类型和方向下的覆盖结果
4. 通过遮挡检测剔除穿墙覆盖
5. 使用贪心策略逐步选取提升覆盖收益最大的点位
6. 计算盲区、重叠区域和建议摄像头数量
7. 输出建议点位、朝向和覆盖范围

### 6.4 人工修正后的重算策略

为了贴合真实工作流，建议支持两种重算模式：

1. 整体重算
   - 适用于覆盖距离改动较大或初次生成方案

2. 保留人工修改后重算
   - 支持锁定已人工调整的摄像头
   - 仅在未锁定区域补点或优化

## 7. 数据模型

每个项目建议至少保存以下信息：

### 7.1 原始图纸

- 原始文件路径或副本
- 文件类型
- 分辨率
- PDF 页码信息

### 7.2 识别结果

- 比例尺或尺寸识别结果
- 墙体线段
- 门洞与开口区域
- 识别置信度
- 待用户确认项

### 7.3 布点参数

- 覆盖距离
- 摄像头类型
- 方向型视场角
- 遮挡规则

### 7.4 布点结果

- 摄像头编号
- 坐标位置
- 朝向角度
- 覆盖图形与范围
- 是否由系统自动生成
- 是否被用户手工修改
- 是否锁定

### 7.5 导出配置

- 是否显示编号
- 是否显示覆盖区域
- 是否显示盲区提示
- 导出格式

建议保存为本地项目文件，例如 `*.camera-plan.json`，用于后续重复打开和继续编辑。

## 8. 编辑能力设计

自动生成结果后，系统应支持以下操作：

- 拖拽摄像头位置
- 旋转方向型摄像头朝向
- 切换单个摄像头类型
- 新增摄像头
- 删除摄像头
- 锁定摄像头
- 修正墙体和门洞
- 调整覆盖距离滑块后重新计算

这些编辑能力是第一版可用性的关键，因为图纸识别和布点算法不可能完全替代人工经验。

## 9. 导出设计

系统需要支持两类成果：

1. 客户交付图
   - `PNG/JPG`
   - `PDF`

2. 内部可编辑项目
   - 本地项目文件

导出图中建议默认包含：

- 原始平面图底图
- 摄像头图标
- 摄像头编号，例如 `CAM-01`
- 覆盖扇形或圆形半透明范围
- 可选的盲区提示

## 10. 错误处理设计

系统需要对关键失败路径做显式提示：

### 10.1 比例识别失败

- 提示未可靠识别比例尺或尺寸标注
- 引导进入人工校准

### 10.2 结构识别失败或低置信度

- 高亮低置信度墙体或门洞
- 允许人工修正后重新计算

### 10.3 自动布点结果不理想

- 提示盲区位置
- 提示重复覆盖过高区域
- 支持整体重算或保留人工修正后重算

### 10.4 导出失败

- 保留当前项目状态
- 提示失败原因，例如图片过大、PDF 转换失败或项目数据异常

## 11. 原型实施备注

这份设计文档对应的第一版原型已经完成落地，当前实现保持了以下边界：

- 后端使用 `FastAPI` 提供项目、布局、识别和导出接口
- 识别层采用 `Pillow + 启发式` 做本地可运行的自动识别，保留低置信度提示
- 布点层使用采样点 + 候选点 + 贪心选点的稳定版本，便于人工修正和后续演进
- 前端工作台支持上传、识别、覆盖距离滑块、锁定相机、重算和导出
- 导出会把项目文件、PNG 和 PDF 一并写入本地 `data/projects/<项目ID>/`

已在真实样例上完成端到端验证，证明以下流程可用：

1. 上传平面图
2. 自动识别比例与墙体建议
3. 手工调整覆盖距离
4. 锁定相机后重算
5. 导出 PNG、PDF 与项目文件

该原型仍然保留“自动建议 + 人工确认”的工作方式，识别和布点都不会替代最终人工审图。

## 11. 测试策略

### 11.1 算法测试

- 覆盖距离变化后摄像头数量变化是否合理
- 方向型与全景型覆盖计算是否正确
- 覆盖不可穿墙、可通过门洞的规则是否正确

### 11.2 交互测试

必须覆盖核心主链路：

- 上传图纸
- 自动识别比例
- 自动识别结构
- 调整滑块
- 自动布点
- 人工拖拽、旋转、增删
- 导出 `PNG/JPG/PDF`

### 11.3 样例验收

使用真实门店图纸进行至少 3 到 5 组案例验证，对比人工经验评估以下指标：

- 推荐数量是否合理
- 点位位置是否接近人工判断
- 盲区提示是否有效
- 导出图是否能直接用于交付

## 12. 第一版边界

第一版最终交付目标包含：

- 支持 `JPG/PNG/PDF`
- 支持方向型和全景型摄像头
- 支持覆盖距离滑动调整
- 支持比例和结构自动识别
- 支持自动布点与人工微调
- 支持导出 `PNG/JPG/PDF`
- 支持项目文件保存与再次编辑

第一版暂不包含：

- `DWG/DXF` 原生解析
- 云端协作
- 多楼层联动
- 基于业务语义的重点区域智能布点
- 完全替代人工判断的最终方案生成

## 13. 推荐实现路线

为了避免“目标范围”和“开发顺序”冲突，建议按里程碑推进，但最终第一版交付仍以上述边界为准。

### 13.1 里程碑一：可用主链路

先实现一条能跑通的主链路：

1. 图纸上传与画布显示
2. 覆盖距离滑块与摄像头类型切换
3. 手工校准比例
4. 基础墙体/门洞人工标注
5. 自动布点算法 MVP
6. 人工微调
7. 导出图片和 PDF

这一阶段的目标是尽快拿真实图纸验证布点算法、交互体验和导出质量。

### 13.2 里程碑二：自动识别补强

在主链路稳定后补强以下能力，使系统达到第一版最终交付目标：

1. 自动识别比例尺或尺寸标注
2. 自动提取墙体和门洞
3. 对低置信度识别结果进行高亮提示
4. 在人工修正后参与自动重算

### 13.3 里程碑三：可用性收敛

1. 完善锁定后重算
2. 优化盲区和重叠提示
3. 用真实门店图做样例验收
4. 优化导出样式，确保可直接交付客户

这样可以更快拿真实图纸做验证，尽早迭代算法和交互，而不是被前期高精度识别卡住整条产品链路。
